## 🔴 КРИТИЧЕСКИЕ проблемы

### 1. TypeScript `strict: true` НЕ включён
**Файл:** `backend/tsconfig.json`
**Конвенция:** «TypeScript 5+ с `strict: true`. Никаких `any` без явного обоснования.»
**Находка:** Несколько строгих флагов явно установлены в `false`:
- `strictNullChecks: false`
- `noImplicitAny: false`
- `strictBindCallApply: false`
- `forceConsistentCasingInFileNames: false`

**Рекомендация:** Заменить все отдельные флаги на `"strict": true`.

---

### 2. Swagger/OpenAPI документация полностью отсутствует
**Конвенция:** «Каждый эндпоинт ОБЯЗАН иметь Swagger-декораторы (`@ApiTags`, `@ApiOperation`, `@ApiResponse`). Каждая DTO ОБЯЗАНА иметь `@ApiProperty`.»
**Находка:**
- `@nestjs/swagger` даже не указан в зависимостях `package.json`
- Ноль декораторов `@ApiTags`, `@ApiOperation`, `@ApiResponse`, `@ApiProperty` во всей кодовой базе
- Swagger не настроен в `main.ts`

**Рекомендация:** Установить `@nestjs/swagger`, добавить декораторы ко всем контроллерам и DTO, настроить `SwaggerModule` в `main.ts`.

---

### 3. `RegisterDto` позволяет произвольное назначение роли
**Файл:** `backend/src/auth/dto/register.dto.ts`
**Находка:** Пользователь может передать `role: 'admin'` при регистрации — никакой валидации или ограничения.
**Влияние:** Любой пользователь может назначить себе права администратора.

**Рекомендация:** Полностью убрать поле `role` из `RegisterDto`; назначать роль `'user'` по умолчанию в логике сервиса.

---

### 4. JWT-секрет откатывается к захардкоженному `'secret'`
**Файл:** `backend/src/auth/auth.module.ts`
**Находка:** `secret: process.env.JWT_SECRET || 'secret'` — если переменная окружения не задана, используется тривиально угадываемый секрет.

**Рекомендация:** Бросать ошибку при запуске, если `JWT_SECRET` не установлен. Никогда не использовать фолбэк для секретов.

---

### 5. `synchronize: true` в конфигурации базы данных
**Файл:** `backend/src/app.module.ts`
**Конвенция:** «В production используются ТОЛЬКО миграции. `synchronize: true` допустим только в development.»
**Находка:** `synchronize: true` установлен безусловно, без проверки окружения. Директория миграций и скрипты отсутствуют.

**Рекомендация:** Установить `synchronize: false`, создать директорию `database/migrations/`, добавить скрипты генерации и запуска миграций.

---

## 🟠 ВЫСОКИЕ проблемы

### 6. Отсутствует паттерн Repository
**Конвенция:** «Controller → Service → Repository → Entity. Один репозиторий на сущность.»
**Находка:** Все сервисы инжектят `Repository<Entity>` из TypeORM напрямую через `@InjectRepository()`. Кастомных репозиториев не существует.

**Рекомендация:** Создать кастомные классы репозиториев (например, `UserRepository`, `AssetRepository`), расширяющие `Repository<Entity>` и инкапсулирующие логику запросов.

---

### 7. Отсутствует директория `src/common/`
**Конвенция:** Ожидается `src/common/` с поддиректориями: `guards/`, `filters/`, `interceptors/`, `pipes/`, `decorators/`.
**Находка:** Директория не существует. Вся общая инфраструктура отсутствует.

**Рекомендация:** Создать `src/common/` со всеми необходимыми поддиректориями и перенести/создать общие компоненты.

---

### 8. Отсутствуют Exception Filters
**Конвенция:** «Глобальный `AllExceptionsFilter` + доменные фильтры.»
**Находка:** Ноль фильтров исключений в кодовой базе.

**Рекомендация:** Создать `AllExceptionsFilter` (глобальный) и доменные фильтры по необходимости.

---

### 9. Отсутствуют Interceptors
**Конвенция:** «Как минимум `LoggingInterceptor` и `TransformInterceptor`.»
**Находка:** Ноль интерсепторов в кодовой базе.

**Рекомендация:** Создать `LoggingInterceptor` (логирование запросов/ответов) и `TransformInterceptor` (обёртка ответов).

---

### 10. Отсутствуют кастомные декораторы (`@Roles()`)
**Конвенция:** Ролевой доступ требует декоратора `@Roles()` + `RolesGuard`.
**Находка:** Ролевой доступ упоминается в бизнес-логике, но декоратор `@Roles()` и `RolesGuard` не существуют.

**Рекомендация:** Создать декоратор `@Roles()` и `RolesGuard` в `src/common/`.

---

### 11. Эндпоинт `resetRoles` не защищён проверкой роли администратора
**Файл:** `backend/src/auth/auth.controller.ts`
**Находка:** Только `@UseGuards(AuthGuard('jwt'))` — любой аутентифицированный пользователь может сбросить все роли в системе.

**Рекомендация:** Добавить гард `@Roles('admin')` (после его создания) к этому эндпоинту.

---

### 12. `findAll()` в Assets возвращает ВСЕ активы
**Файл:** `backend/src/assets/assets.service.ts`
**Находка:** Возвращает все активы в базе данных без фильтрации по запрашивающему пользователю. Потенциальная утечка данных между пользователями.

**Рекомендация:** Фильтровать по `userId` из JWT-токена.

---

### 13. Отсутствуют юнит-тесты для критических сервисов
**Находка:** Следующие сервисы НЕ имеют файлов юнит-тестов:
- `notifications.controller.spec.ts` ❌
- `notifications.service.spec.ts` ❌
- `notification.service.spec.ts` ❌
- `email.service.spec.ts` ❌
- `scheduler.service.spec.ts` ❌

**Рекомендация:** Создать полноценные юнит-тесты для всех сервисов.

## 🟡 СРЕДНИЕ проблемы

### 14. Отсутствует родительская директория `src/modules/`
**Конвенция:** Модули должны находиться в `src/modules/`.
**Находка:** Модули расположены на корневом уровне `src/` (`src/auth/`, `src/assets/` и т.д.).

**Рекомендация:** Перенести модули в директорию `src/modules/`.

---

### 15. Поле Role имеет тип `string`
**Файл:** `backend/src/auth/user.entity.ts`
**Находка:** `role` типизирован как `string` вместо enum.

**Рекомендация:** Создать enum `UserRole` (`user`, `admin`) и использовать его для поля сущности и всех связанных DTO.

---

### 16. `@Request() req` имеет тип `any`
**Находка:** Все методы контроллеров, использующие `@Request() req`, не имеют аннотации типа — `req` неявно является `any`.

**Рекомендация:** Создать типизированный интерфейс (например, `AuthenticatedRequest`), расширяющий `Request` со свойством `user`.

---

### 17. `UpdateUserSettingsDto` не использует `PartialType`
**Файл:** `backend/src/user-settings/core/dto/update-user-settings.dto.ts`
**Конвенция:** «Используйте `PartialType(CreateDto)` для update DTO.»
**Находка:** Вручную переобъявляет все поля вместо наследования от `CreateUserSettingsDto`.

**Рекомендация:** Использовать `export class UpdateUserSettingsDto extends PartialType(CreateUserSettingsDto) {}`.

---

### 18. Два сервиса уведомлений с запутанными именами
**Файлы:** `notifications.service.ts` И `notification.service.ts`
**Находка:** Неясное разделение ответственности, путаница в именовании.

**Рекомендация:** Объединить или чётко переименовать с разграничением ответственности.

---

### 19. Не настроен порог покрытия тестами
**Конвенция:** «Покрытие минимум 80%.»
**Находка:** В конфигурации Jest отсутствует `coverageThreshold`.

**Рекомендация:** Добавить `coverageThreshold: { global: { branches: 80, functions: 80, lines: 80, statements: 80 } }` в конфигурацию Jest.

---

### 20. Отсутствует `enableShutdownHooks()` в main.ts
**Конвенция:** «Рекомендуется для корректного завершения работы.»
**Находка:** `main.ts` не вызывает `app.enableShutdownHooks()`.

**Рекомендация:** Добавить `app.enableShutdownHooks()` перед `app.listen()`.
