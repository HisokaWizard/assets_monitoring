# AGENTS.md — Мультиагентная система разработки

Все настройки агента хранятся в `.opencode/`.

---

## 1. Обзор приложения

Система мониторинга криптовалютных активов и NFT с автоматическими обновлениями, уведомлениями и отчетами.

**Ключевой функционал:**

- **Мониторинг** — автоматическое обновление цен (CoinMarketCap для крипто, OpenSea для NFT) каждые 4 часа
- **Персонализация** — настройка интервалов обновления (2-12 часов) и порогов срабатывания алертов
- **Уведомления** — email-информирование при достижении заданного % изменения цены
- **Аналитика** — генерация отчетов (день, неделя, месяц, квартал, год) и хранение истории цен
- **Безопасность** — JWT-аутентификация (роли: `user`, `admin`)

**Стек:**

| Слой | Технологии |
|------|-----------|
| Backend | NestJS, TypeORM, PostgreSQL, Jest |
| Frontend | React 18+, Redux (RTK), MUI, Playwright, FSD-архитектура |
| Общее | TypeScript 5+ (`strict: true`), documents_hub как источник конвенций |

## 2. Ключевые принципы разработки

### 2.1. Строгая типизация

TypeScript 5+ с `strict: true`. Никаких `any` без явного обоснования. Все публичные методы и классы документируются через JSDoc.

### 2.2. Стиль кода

- 2 пробела для отступов
- Одинарные кавычки
- Точки с запятой обязательны
- Информативные имена переменных и функций

### 2.3. TDD (Test-Driven Development)

Любой код пишется по циклу Red-Green-Refactor: сначала падающий тест, затем минимальная реализация, затем рефакторинг. Правила тестирования для каждого слоя читаются из `documents_hub`.

### 2.4. Архитектурное разделение

- **Backend** — многослойная архитектура (Controller -> Service -> Repository -> Entity). Конвенции: `documents_hub/backend/`
- **Frontend** — FSD (Feature-Sliced Design): `app/ -> pages/ -> widgets/ -> features/ -> entities/ -> shared/`. Конвенции: `documents_hub/frontend/`

### 2.5. Изоляция слоев

Backend и frontend — независимые части системы. Каждая имеет свою нумерацию задач, свои конвенции, свои тесты. Общение между ними только через REST API.

### 2.6. Автоматическая документация

Документация сервиса должна автоматически обновляться при изменениях в репозитории. Требования:

- **API-документация** — генерируется автоматически из Swagger/OpenAPI-декораторов в NestJS-контроллерах. При каждом изменении эндпоинта документация обновляется автоматически
- **Архитектурная документация** — хранится в `docs/` в корне проекта и обновляется при структурных изменениях (добавление модулей, изменение схемы БД, новые интеграции)
- **Changelog** — ведется автоматически на основе conventional commits. Каждый PR должен содержать описание изменений
- **Компонентная документация (frontend)** — Storybook для UI-компонентов, обновляется при изменении компонентов в `shared/ui/` и `widgets/`
- **Схема БД** — ER-диаграмма генерируется из TypeORM-сущностей и обновляется при изменении entity-файлов

При создании или модификации любой фичи, агент-ревьювер проверяет актуальность документации и при необходимости создает задачу на обновление.

## 3. Мультиагентная система

Вся разработка ведется через систему из 6 специализированных агентов. Конфигурация агентов — в `opencode.json` (блок `"agent"`), промпты — в `.opencode/prompts/*.txt`.

### 3.1. Агенты и их роли

| Агент | Тип | Промпт | Роль |
|-------|-----|--------|------|
| **architect** | `primary` | `.opencode/prompts/architect.txt` | Оркестратор. Принимает задачи, координирует саб-агентов, обновляет STATUS.md. Не пишет код. |
| **planner** | `subagent` | `.opencode/prompts/planner.txt` | Планировщик. Собирает контекст из `contextDocs`, декомпозирует на 3-5 sub_task, создает файлы в `.opencode/features/`. |
| **tester** | `subagent` | `.opencode/prompts/tester.txt` | Тестировщик. TDD: Red (тесты падают), Green (тесты проходят), Validation (регрессия + coverage 80%). |
| **backend-dev** | `subagent` | `.opencode/prompts/backend-dev.txt` | Backend-разработчик. NestJS, TypeORM, JWT. Реализует код по готовым тестам. |
| **frontend-dev** | `subagent` | `.opencode/prompts/frontend-dev.txt` | Frontend-разработчик. React 18+, Redux Toolkit, MUI, FSD. Реализует код по готовым тестам. |
| **code-reviewer** | `subagent` | `.opencode/prompts/code-reviewer.txt` | Ревьювер. Проверяет архитектуру, типизацию, безопасность, документацию. Approve / Request changes. |

### 3.2. Порядок оркестрации

```
Пользователь -> architect -> @planner (декомпозиция)
                    |
                    +-> Подтверждение от пользователя
                    |
                    +-> Для каждого sub_task:
                    |       @tester (RED) -> @backend-dev/@frontend-dev (код) -> @tester (GREEN) -> @code-reviewer
                    |       [При проблемах: цикл исправлений, max 5 итераций]
                    |
                    +-> STATUS.md -> "done"
```

### 3.3. Структура конфигурации

```
opencode.json                    # Декларация всех агентов (блок "agent")
.opencode/
├── prompts/                     # Изолированные промпт-файлы
│   ├── architect.txt
│   ├── planner.txt
│   ├── tester.txt
│   ├── backend-dev.txt
│   ├── frontend-dev.txt
│   └── code-reviewer.txt
├── skills/                      # Скиллы агентов
│   ├── debugger/SKILL.md
│   └── tdd-workflow/SKILL.md
├── tools/
│   └── contextDocs.ts           # Кастомный тул для documents_hub
└── features/                    # Хранилище задач
    ├── backend/feat_1..feat_N/
    └── frontend/feat_1..feat_N/
```

## 4. Documents Hub — внешний источник конвенций

### 4.1. Описание

`documents_hub` — централизованный репозиторий с правилами, конвенциями и гайдлайнами для всех слоев приложения. Это единственный источник истины для архитектурных решений.

### 4.2. Скилл работы с documents_hub

Для обращения к documents_hub используется tool `contextDocs`. Каждый агент ОБЯЗАН загружать релевантные конвенции перед началом работы.

**API tool `contextDocs`:**

```
contextDocs(layer, topic)       — получить документ по теме
contextDocs(layer, search)      — поиск по ключевому слову
contextDocs(layer, section)     — получить конкретную секцию
contextDocs(layer, listSections) — список доступных секций
```

**Примеры вызовов:**

```
contextDocs(layer: "backend", listSections: true)                    — список всех backend-секций
contextDocs(layer: "backend", topic: "NEST_JS_ARCHITECTURE")         — архитектура NestJS
contextDocs(layer: "backend", topic: "BACKEND_TESTING")              — правила тестирования backend
contextDocs(layer: "frontend", topic: "FSD")                         — FSD-архитектура
contextDocs(layer: "frontend", topic: "TESTING")                     — правила тестирования frontend
contextDocs(layer: "backend", search: "repository pattern")          — поиск по ключевому слову
```

**Правило:** Если агент не загрузил контекст из documents_hub перед началом работы — это нарушение процесса. Архитектор должен остановить выполнение и потребовать загрузки контекста.

### 4.3. Tool для синхронизации documents_hub

Для подтягивания репозитория documents_hub на локальную машину используется tool, который:

1. Определяет путь к хабу из переменной окружения `DOCUMENTS_HUB_PATH`
2. Если переменная не задана — используется путь `../documents_hub` (относительно корня проекта)
3. Если локальная копия отсутствует — клонирует из remote:
   `https://github.com/HisokaWizard/documents_hub`
4. Raw-контент доступен по:
   `https://raw.githubusercontent.com/HisokaWizard/documents_hub/main`

**Приоритет источников:**

```
1. Локальная копия по $DOCUMENTS_HUB_PATH (если задана)
2. Локальная копия по ../documents_hub (дефолт)
3. GitHub raw content (fallback)
```

**Команда синхронизации:**

```bash
# Если папка существует — pull
git -C "$DOCUMENTS_HUB_PATH" pull origin main

# Если папки нет — clone
git clone https://github.com/HisokaWizard/documents_hub.git "$DOCUMENTS_HUB_PATH"
```

## 5. Бизнес-логика приложения

### 5.1. Управление активами

Для каждого актива (криптовалюта или NFT) в портфеле пользователя система:

1. Опрашивает внешние API (CoinMarketCap, OpenSea)
2. Рассчитывает % изменения к текущей цене
3. Логирует исторические данные для графиков

### 5.2. Логика уведомлений

Алерты срабатывают на базе фильтров пользователя:

- **Порог изменения** — уведомлять только при скачке > N% (настраивается)
- **Интервал** — ограничение частоты писем (раз в 2-12 часов) для избежания спама

### 5.3. Алгоритм отчетов

Отчеты (Daily, Weekly, Monthly, Quarterly, Yearly) рассчитываются как дельта между ценой на начало периода и текущей рыночной ценой. По завершении периода поля обновляются новыми значениями.

## 6. Процесс работы с задачами

### 6.1. Полный цикл задачи

```
1. Пользователь ставит задачу агенту-архитектору
2. Архитектор запускает агента-планировщика
3. Планировщик:
   a. Собирает контекст (contextDocs)
   b. Задает уточняющие вопросы (mcp_question)
   c. Декомпозирует задачу на sub_tasks
   d. Создает файлы в .opencode/features/
   e. Обновляет todo-лист (mcp_todowrite)
4. Архитектор представляет план пользователю
5. ОЖИДАНИЕ ПОДТВЕРЖДЕНИЯ ОТ ПОЛЬЗОВАТЕЛЯ
6. Архитектор обновляет STATUS.md -> "in progress"
7. Для каждого sub_task:
   a. Тестировщик пишет тесты (Red)
   b. Тестировщик запускает тесты — все должны УПАСТЬ
   c. Писатель кода реализует решение
   d. Тестировщик запускает тесты — все должны ПРОЙТИ
   e. Ревьювер проверяет код
   f. При проблемах — цикл исправлений (max 5 итераций)
   g. Архитектор отмечает sub_task как выполненный
8. Архитектор обновляет STATUS.md -> "done"
9. Архитектор отчитывается пользователю
```

### 6.2. Формат STATUS.md

```markdown
# Status: feat_{N}

- **Current:** to do | in progress | done | closed
- **Started:** YYYY-MM-DD HH:MM
- **Completed:** YYYY-MM-DD HH:MM
- **Current Sub-task:** sub_task_1.md

## Прогресс

- [ ] sub_task_1.md
- [ ] sub_task_2.md
- [ ] sub_task_3.md
```

### 6.3. Правила работы с задачами

1. **Независимость** — backend и frontend имеют независимую нумерацию задач
2. **Неизменность** — sub_tasks не меняются в процессе работы. Нужны изменения — создается новая задача
3. **Максимум** — не более 5 sub_tasks на одну задачу
4. **Обязательность** — каждая задача начинается с создания структуры в `.opencode/features/`
5. **Контекст** — обязательно использовать `contextDocs` для получения архитектурной информации
6. **Статус** — STATUS.md обновляется при каждом изменении состояния
7. **TDD** — каждый sub_task ДОЛЖЕН содержать тесткейсы. Тесты пишутся ДО реализации
8. **Подтверждение** — НИКОГДА не приступать к выполнению без подтверждения от пользователя
9. **Документация** — при изменениях проверять актуальность документации сервиса

## 7. Автоматическая документация сервиса

### 7.1. Типы документации

| Тип | Инструмент | Триггер обновления |
|-----|-----------|-------------------|
| API-документация | Swagger/OpenAPI декораторы в NestJS | Изменение контроллеров/DTO |
| Архитектурная документация | Markdown в `docs/` | Добавление модулей, изменение схемы БД |
| Changelog | Conventional commits | Каждый PR/коммит |
| Компонентная документация | Storybook | Изменение компонентов в `shared/ui/`, `widgets/` |
| Схема БД | ER-диаграмма из TypeORM entities | Изменение entity-файлов |

### 7.2. Правила поддержания документации

- Каждый эндпоинт ОБЯЗАН иметь Swagger-декораторы (`@ApiTags`, `@ApiOperation`, `@ApiResponse`)
- Каждая DTO ОБЯЗАНА иметь `@ApiProperty` декораторы
- При добавлении нового модуля — обновить архитектурную документацию в `docs/`
- При изменении entity — проверить и обновить ER-диаграмму
- Агент-ревьювер проверяет актуальность документации на каждом ревью
- Коммиты следуют формату Conventional Commits для автоматической генерации changelog

### 7.3. Структура документации

```
docs/
├── architecture/
│   ├── overview.md            # Общая архитектура системы
│   ├── backend-modules.md     # Описание backend-модулей
│   └── frontend-structure.md  # Описание frontend-структуры
├── api/
│   └── (генерируется автоматически из Swagger)
├── database/
│   ├── er-diagram.md          # ER-диаграмма
│   └── migrations-log.md      # Лог миграций
└── changelog.md               # Автоматический changelog
```
