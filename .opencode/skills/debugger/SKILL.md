# Debugger — Систематическая отладка и поиск первопричин

> **name:** debuggers
> **description:** Систематическая отладка и анализ первопричин для выявления и исправления программных ошибок. Используй, когда: отлаживаешь ошибки, устраняешь баги, расследуешь падения, анализируешь стек-трейсы, чинишь сломанный код, или когда пользователь упоминает debugging, error, bug, crash, "не работает".

---

Ты — эксперт по отладке, который использует систематический подход для эффективного выявления и устранения программных ошибок.

---

## Когда применять

Используй этот навык, когда:

- Расследуешь баги или неожиданное поведение
- Анализируешь сообщения об ошибках и стек-трейсы
- Устраняешь проблемы с производительностью
- Отлаживаешь инциденты в продакшене
- Ищешь первопричины сбоев
- Анализируешь crash-дампы или логи
- Решаешь проблемы, возникающие периодически

---

## Процесс отладки

Следуй этому систематическому подходу:

### 1. Пойми проблему

- Какое **ожидаемое** поведение?
- Какое **фактическое** поведение?
- Можно ли воспроизвести проблему **стабильно**?
- **Когда** это начало происходить?
- **Что изменилось** в последнее время?

### 2. Собери информацию

- Сообщения об ошибках и стек-трейсы
- Лог-файлы и журналы ошибок
- Детали окружения (ОС, версии, конфигурация)
- Входные данные, которые вызывают проблему
- Состояние системы до / во время / после

### 3. Сформулируй гипотезы

- Каковы **наиболее вероятные** причины?
- Перечисли гипотезы от наиболее вероятной к наименее
- Рассмотри: логические ошибки, проблемы с данными, окружение, тайминг, зависимости

### 4. Проверь гипотезы

- Используй **бинарный поиск** для сужения области
- Добавь логирование / print-выводы в стратегических точках
- Используй точки останова (breakpoints)
- Изолируй компоненты
- Проверь на минимальном воспроизводимом примере

### 5. Найди первопричину

- **Не останавливайся** на симптомах — найди реальную причину
- Подтверди доказательствами
- Пойми, почему это не было обнаружено раньше

### 6. Исправь и проверь

- Реализуй исправление
- Тщательно протестируй фикс
- Убедись в отсутствии регрессий
- Добавь тесты для предотвращения повторения

---

## Стратегии отладки

### Бинарный поиск

1. Определи область кода (начало → конец)
2. Проверь **середину**
3. Если баг присутствует → ищи в **левой** половине
4. Если баг отсутствует → ищи в **правой** половине
5. Повторяй до изоляции проблемы

### Метод "Резиновой уточки"

- Объясни код **строка за строкой** вслух
- Часто это выявляет проблему через вербализацию
- Проясняет скрытые допущения

### Стратегическое логирование

```python
# На входе в функцию
print(f"[DEBUG] function_name called with: {args}")

# В точках принятия решений
print(f"[DEBUG] Condition X is {condition_result}")

# До и после изменения состояния
print(f"[DEBUG] Before: {state}, After: {new_state}")
```

### Метод Bisect (для регрессий)

```bash
# Найди коммит, который привнёс баг
git bisect start
git bisect bad HEAD
git bisect good <последний-хороший-коммит>

# Тестируй каждую ревизию, пока не найдёшь
```

## Распространённые паттерны багов

| Паттерн                                | Описание                                                                                       |
| -------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **Ошибка на единицу (Off-by-One)**     | Индексы цикла (`i < n` vs `i <= n`), выход за границы массива (`arr[len]` вместо `arr[len-1]`) |
| **Null / Undefined ссылки**            | Проверяй переменные перед использованием, убедись что ответы API содержат ожидаемые поля       |
| **Состояние гонки (Race Condition)**   | Асинхронные операции завершаются в неожиданном порядке, общее состояние без блокировок         |
| **Несоответствие типов**               | Сравнение строки с числом, неявное приведение типов                                            |
| **Устаревшее состояние (Stale State)** | Кэшированные значения не обновлены, замыкание захватывает старые переменные                    |

## Пример

**Жалоба пользователя:** "Мой API случайным образом возвращает ошибки 500"

---

### Описание проблемы

API-эндпоинт периодически возвращает HTTP 500. Не все запросы падают — часть проходит успешно.

### Окружение

- Версия Node.js?
- Тип базы данных?
- Частота запросов / конкурентность?

### Анализ ошибки

Необходимо получить:

- Полный стек-трейс ошибки
- Серверные логи в момент сбоя
- Пример запроса, который падает

### Гипотезы (по приоритету)

1. **Исчерпание пула подключений к БД** — периодический характер указывает на конкуренцию за ресурсы
   - Проверить: корреляция сбоев с высокой нагрузкой
   - Проверить: размер пула соединений vs количество конкурентных запросов

2. **Не дождались асинхронной операции** — состояние гонки в коде
   - Проверить: использование `async/await`
   - Проверить: пропущенные ключевые слова `await`

3. **Необработанное отклонение промиса** — часть путей ошибок не обработана
   - Проверить: промисы без `.catch()` или `try/catch`

### Шаги расследования

**1. Добавить детальное логирование:**

```javascript
app.post('/api/endpoint', async (req, res) => {
  console.log('[DEBUG] Получен запрос:', req.body);
  try {
    const result = await someOperation();
    console.log('[DEBUG] Операция выполнена успешно');
    res.json(result);
  } catch (error) {
    console.error('[ERROR] Операция упала:', error.stack);
    res.status(500).json({ error: error.message });
  }
});
```

**2. Мониторинг пула подключений:**

```javascript
db.on('acquire', () => {
  console.log(`[POOL] Соединение получено (${db.pool.size}/${db.pool.max})`);
});
```

**3. Ловить необработанные отклонения:**

```javascript
process.on('unhandledRejection', (reason, promise) => {
  console.error('[FATAL] Необработанное отклонение промиса:', reason);
});
```

### Дальнейшие шаги

Задеплоить изменения с логированием и отслеживать паттерны по:

- Времени суток
- Конкретным данным пользователя
- Использованию ресурсов сервера (CPU, память, соединения)
